# 需求文档规范

## 1. 文档结构
每个功能模块的需求文档（如 `docs/<模块>/需求.md`）应包含以下章节：

### 1.1 背景与目标
- 简述为何需要此功能，解决了什么问题。
- 定义核心用户角色。

### 1.2 用户故事 (User Stories)
- 格式：**作为一个[角色]，我想要[功能]，以便[目的]。**
- *示例*：“作为一个管理员，我想要禁用某个应用，以便阻止用户访问有问题的服务。”

### 1.3 功能详情 (Functional Requirements)
详细描述功能的具体行为。对于复杂逻辑，应包含：
- **输入/输出**：需要填写什么字段，系统返回什么。
- **流程图/时序图**：(可选) 使用 Mermaid 语法描述流程。
- **状态流转**：如有状态变化（如订单状态），需定义状态机。

### 1.4 数据模型 (Data Model)
- 列出涉及的核心实体。
- 定义字段属性：名称、类型、是否必填、默认值、校验规则。

### 1.5 交互与界面 (UX/UI)
- **原型位置**：所有页面布局草图（ASCII/线框图）或交互描述必须包含在模块目录下的 **页面设计文档**（如 `docs/<模块>/页面设计.md`）中。
- **素材存储**：如果是截图、流程图图片等二进制素材，必须存放于 `docs/<模块>/assets/` 子目录。
- **内容要求**：
    - 页面布局草图。
    - 关键交互说明（如：点击删除弹出二次确认）。

### 1.6 验收标准 (Acceptance Criteria)
- 列出“完成”的定义 (DoD)。
- 明确的测试点，包括正常路径和异常路径。

## 2. 编写原则
- **明确性**：避免使用“可能”、“大概”、“最好支持”等模糊词汇。
- **原子性**：每个需求点应独立且可测试。
- **完整性**：必须包含边界条件和异常处理说明。

## 3. 维护规范
- **单一事实来源 (SSOT)**：代码实现以文档为准。
- **变更记录**：文档开头应记录变更历史（日期、修改人、变更内容）。
